Project Overview

1.1 What the system does

The Finance AI Agent answers natural language questions about Indian equities — specifically HDFC Bank and Reliance Industries — by intelligently routing each question to the right data source and synthesizing a structured, grounded answer.

A user can ask questions like:

  • "What was HDFC Bank's PAT in Q3 FY26?" → fetches from indexed earnings PDFs
  • "Why did Reliance stock fall on 2026-01-15?" → fetches price data + news articles
  • "What is HDFC's current P/E ratio?" → fetches live market fundamentals
  • "Summarize management commentary from Q4 FY26 earnings" → fetches from RAG documents

1.2 Why we built it this way

The core problem with financial Q&A is hallucination — LLMs trained on general data confidently state wrong numbers. Our design solves this in three ways:

  • 1
    Never answer from memory. The agent always calls a tool. Tools either return grounded document context (RAG), live market data (yfinance), or real news (GNews). The LLM only synthesizes — it never invents.
  • 2
    Hybrid retrieval + reranking. Using both dense vector search and BM25 keyword search, then cross-encoder reranking, maximizes the chance of pulling the exact chunk that contains the answer.
  • 3
    Strict generation prompt. The RAG generation step explicitly prohibits the LLM from using outside knowledge and mandates citing sources.

2.2 Why each key tool was chosen

Gemini 2.5 Flash — not GPT-4 or Claude

Gemini 2.5 Flash has native function/tool-calling support in the Google GenAI SDK, extremely low latency, and a large context window (1M tokens). For a financial assistant that needs to reason about long document context and call multiple tools in sequence, Flash is significantly faster and cheaper than Opus-class models while maintaining quality.

LlamaIndex — not LangChain

LlamaIndex is purpose-built for RAG and document-based question answering. It provides built-in VectorStoreIndex, BM25Retriever, QueryFusionRetriever, and FlagEmbeddingReranker — all the pipeline components we need, composable and well-integrated. LangChain requires more manual assembly for equivalent RAG quality.

all-MiniLM-L6-v2 — local embeddings

This 22M-parameter model runs locally with no API cost, generates 384-dimensional vectors, and is fast (~50ms per batch). For a project querying PDFs at query time, local embeddings avoid API latency and cost. The model is well-tested on English financial text.

BAAI/bge-reranker-base — cross-encoder reranker

A bi-encoder (like MiniLM) compresses each document into a single vector — it loses nuance. A cross-encoder reads the query and document together, capturing token-level interactions. BGE-reranker-base is small (278M params) yet dramatically improves precision by re-scoring each retrieved chunk against the full query context.

FastAPI — not Flask or Django

FastAPI is natively async, which matters because our /ask endpoint awaits an async agent (agent.run()) and streams events. Flask is synchronous by default; Django is too heavy for a single-endpoint API.

Data Ingestion & Index Building

Before any query can be answered from documents, the PDFs must be processed and stored in a searchable format. This is a one-time (or incremental) operation run via build_index.py.

3.1 Document loading

LlamaIndex's SimpleDirectoryReader loads all files from data/{company}/, including nested folders. The filename_as_id=True flag sets each document's ID to its file path, enabling incremental updates — if a PDF is replaced, the old version is removed and the new one re-embedded.

documents = SimpleDirectoryReader(
    input_dir="data/hdfc",
    recursive=True,
    filename_as_id=True   # enables incremental refresh
).load_data()

3.2 Chunking strategy

Long PDFs cannot be searched as a whole — they must be split into smaller pieces called chunks (also called nodes). The SentenceSplitter splits on sentence boundaries to avoid cutting mid-sentence.

ParameterValueWhy
chunk_size512 tokensFits dense financial text while staying within retriever scoring limits
chunk_overlap100 tokensOverlapping windows ensure numbers that span sentence boundaries are not split between two non-overlapping chunks
Known limitation: Financial PDFs contain tables where a single row might span 600+ tokens. A 512-token chunk can cut through a table row, splitting a number from its label. Increasing chunk_size to 768 and chunk_overlap to 150 would improve retrieval accuracy for tabular data.

3.3 Embedding & vector store

Each chunk is passed through all-MiniLM-L6-v2 to produce a 384-dimensional embedding vector. These vectors — along with the raw chunk text and metadata (filename, page number) — are stored in a VectorStoreIndex and persisted to disk at vector_store/{company}/.

After persisting the vector index, build_index.py also saves the node corpus as JSON so BM25 can be rebuilt quickly on demand. Both artifacts are built once and reused at query time:

FileContentsUsed by
vector_store.json384-dim embedding vectors for all chunksVector retriever (similarity search)
docstore.jsonRaw chunk text + metadata (filename, page)BM25 build, context assembly, reranker
index_store.jsonIndex metadata and node mappingsLlamaIndex internal bookkeeping
bm25_nodes.jsonNode text + metadata for all chunks (plain JSON)BM25 rebuild at query time

3.4 Incremental indexing

On subsequent runs, if the vector_store/{company}/ folder already exists, the script calls index.refresh_ref_docs() instead of rebuilding from scratch. This only re-embeds new or changed files:

# Only re-embeds new/changed files
refreshed = index.refresh_ref_docs(documents)
new_count = sum(refreshed)

# Remove PDFs that were deleted from data/
deleted_ids = indexed_doc_ids - current_doc_ids
for doc_id in deleted_ids:
    index.delete_ref_doc(doc_id, delete_from_docstore=True)

This means adding a new quarterly PDF (e.g. Q4 FY26) only re-embeds that one file, not the entire corpus.

RAG Pipeline (Core)

When the agent calls rag_tool, a 6-step pipeline runs to retrieve relevant document chunks and generate a grounded answer. This is the most technically sophisticated part of the system.

4.1 Query parsing

The agent passes a structured multi-line string to every tool. The parser extracts five fields:

Date: 2026-02-20
Company: HDFC
Financial Year: 2025-26
Quarter: Q3
Question: What was HDFC Bank's PAT in Q3 FY26?

Company names are lower-cased to match the vector_store/hdfc/ folder name.

4.2 Query enrichment

The raw question is enriched with quarter and financial year hints before retrieval. This biases both the vector search and BM25 toward chunks that explicitly mention the target period:

# Before: "What was PAT in Q3 FY26?"
# After:  "What was PAT in Q3 FY26? (Related to quarter 3 (Q3)) (For financial year 2025-26)"

The same enriched query is passed to the reranker so the cross-encoder scores chunks against the full period-aware context, not just the bare question.

4.3 Hybrid retrieval — Vector + BM25

We use two fundamentally different retrieval methods and combine them. Each has strengths the other lacks:

MethodHow it worksGood atWeak at
Vector search Computes cosine similarity between query embedding and chunk embeddings Semantic meaning — finds paraphrases and related concepts Exact terms, numbers, ticker symbols
BM25 Term frequency / inverse document frequency scoring over exact words Exact keywords — company names, metric names, "NIM", "GNPA", "PAT" Semantic similarity, synonyms

For financial queries, both matter. "Net interest margin" should match a chunk that says "NIM" (keyword) and a chunk that discusses "spread between lending and borrowing rates" (semantic). Each retriever fetches the top-15 candidates independently.

4.4 Reciprocal Rank Fusion (RRF)

After the two retrievers return 15 chunks each, they are merged into a single ranked list using Reciprocal Rank Fusion. RRF does not require the scores from both systems to be on the same scale — it only uses rank positions.

The formula for each chunk is: RRF score = Σ 1 / (k + rank_i) where k = 60 (a smoothing constant) and rank_i is the chunk's rank in each retriever. A chunk ranked 1st by both retrievers scores much higher than one ranked 1st by only one.

4.5 Cross-encoder reranking

RRF gives us 15 fused candidates. We then pass all 15 to a cross-encoder reranker (BAAI/bge-reranker-base) that scores each (enriched_query, chunk) pair by reading them together — much more accurate than embedding similarity.

The reranker selects the top-6 most relevant chunks. These 6 chunks form the context passed to the generation step.

Why this matters: Without reranking, a vector search might return the Q2 PAT figure when Q3 is requested, because the embeddings are similar. The cross-encoder, reading "Q3 FY26" in the query vs "Q2 FY26" in the chunk, scores that chunk low and promotes the correct Q3 chunk instead.

4.6 Grounded generation

The top-6 chunks are assembled into a context string with source metadata, then passed to Gemini with a strict prompt:

STRICT INSTRUCTIONS:
- Use ONLY the provided context.
- Do NOT use outside knowledge.
- Do NOT infer numerical values unless explicitly mentioned.
- If the answer is not found, respond: "Not found in documents."
- Cite sources: (Source: file_name, Page: page_number)

Temperature is set to 0.1 for the generation call, making responses deterministic and reducing creative hallucination.

4.7 BM25 persistence & caching

BM25 is built by tokenizing every chunk in the index and computing term frequencies — an O(n) operation over potentially hundreds of chunks. We solve this at two levels:

Level 1 — Disk persistence (build_index.py)

Just like vector embeddings are saved to vector_store/{company}/, the node corpus is saved to vector_store/{company}/bm25_nodes.json every time build_index.py runs. Each entry is plain JSON — chunk text, metadata, and node ID. This avoids rebuilding BM25 from raw PDFs at query time.

Why JSON and not pickle? The BM25Retriever uses pystemmer internally, a Cython library whose objects cannot be serialized with Python's standard pickle. Storing raw node data as JSON sidesteps this entirely — the data is pure Python, always serializable, and BM25 rebuilds from it in under a second.

# In build_index.py — runs once when indexing documents
nodes_data = [
    {"id_": node.node_id, "text": node.text, "metadata": node.metadata}
    for node in index.docstore.docs.values()
]
with open("vector_store/hdfc/bm25_nodes.json", "w") as f:
    json.dump(nodes_data, f)   # saved alongside embeddings

Level 2 — In-memory cache (rag_model.py)

At query time, on the first request per company, the JSON file is loaded and a BM25Retriever is rebuilt from the node data. Every subsequent request reuses the in-memory instance — no disk I/O, no rebuild:

if company not in _bm25_cache:
    with open(f"vector_store/{company}/bm25_nodes.json") as f:
        nodes_data = json.load(f)
    nodes = [TextNode(text=n["text"], metadata=n["metadata"], id_=n["id_"]) for n in nodes_data]
    _bm25_cache[company] = BM25Retriever.from_defaults(nodes=nodes, similarity_top_k=15)
bm25_retriever = _bm25_cache[company]   # instant on every request after first
Comparison with embeddings: Embeddings are built in build_index.py and loaded from disk at query time by LlamaIndex automatically. BM25 now follows the same pattern — build once, persist to disk, reconstruct on demand. The only difference is JSON vs. LlamaIndex's own binary format.

Agent System

5.1 What is a FunctionAgent?

A FunctionAgent (from LlamaIndex) is an LLM that can call functions (tools) to gather information before formulating a response. It works in a ReAct loop: Reason → Act → Observe → Reason again.

1. Receive query
Agent reads the full structured input
2. Reason
Gemini reads system prompt + query, decides which tool to call
3. Call tool
Agent calls rag_tool, fundamental_tool, etc.
4. Observe
Tool result is added to conversation context
5. Repeat or answer
Agent either calls another tool or formulates final answer

The loop is capped at 10 tool calls per query to prevent infinite chains on ambiguous inputs.

5.2 System prompt design

The system prompt is the agent's "constitution" — it defines behavior that applies to every single query. Key sections:

SectionPurpose
Critical RulesHard constraints: never answer from memory, never fabricate numbers, always pass full structured input to tools
Tool Selection GuideExplicit mapping of question types to tool names — reduces ambiguous tool selection
When Data Is Not FoundInstead of just "data not available", the agent must explain which tools were called, why data might be missing, and suggest alternatives
Final Response FormatMandates: (1) Data Summary, (2) Analytical Interpretation, (3) Conclusion — consistent structured output

5.3 Tool routing logic

The agent uses both the system prompt's tool selection guide and each tool's description string to decide which tool to call. Tool descriptions are written to be mutually exclusive — each has a clear "USE FOR" and "DO NOT USE FOR" section:

# Example: agent sees this description for rag_tool:
"USE THIS TOOL FOR:
- Exact financial figures: revenue, PAT, NIM, GNPA, EPS...
- Management commentary, earnings discussion, strategic guidance
- Period-specific questions tied to a quarter or financial year

DO NOT use for:
- Live stock price or intraday data
- Live valuation ratios (P/E, P/B) — use fundamental_tool
- Recent news — use get_gnews_articles"

This explicit boundary-setting prevents the agent from using rag_tool for live P/E ratios (where it would find no data) or fundamental_tool for historical quarter figures (where it would return current-only data).

LLM temperature = 0.1: The agent LLM (Gemini 2.5 Flash) is initialized with temperature=0.1. Low temperature means the model consistently makes the same tool selection for the same query type, reducing variance in routing behavior.

The Four Tools

rag_tool Document RAG Tool

Data source: Indexed earnings PDFs, press releases, investor presentations stored in vector_store/{company}/

What it answers: Any question about a specific quarter or financial year — PAT, NIM, GNPA, CASA ratio, capital adequacy, deposits, advances, management outlook, earnings highlights, segment performance.

Pipeline: Parse → Enrich query → Hybrid retrieval (Vector + BM25) → RRF Fusion → Cross-encoder reranking → Gemini grounded generation

Returns: A grounded prose answer with source citations + a [SOURCES_USED: ...] marker parsed by the evaluation harness.

Example:

Input:  "What was HDFC Bank's PAT in Q3 FY26?"
Output: "HDFC Bank reported a PAT of ₹16,736 crore for Q3 FY2025-26,
         reflecting a 2.2% YoY growth. (Source: Q3FY26-earnings-presentation.pdf, Page: 4)"
fundamental_tool Fundamentals Tool

Data source: Yahoo Finance via yfinance library — live market data

What it answers: Current valuation ratios and company profile metrics: Market Cap, Trailing P/E, Forward P/E, Price-to-Book, Dividend Yield, Beta, ROE, Profit Margins, Sector, Industry.

Important limitation: Returns current/latest data only. Cannot answer "What was the P/E in Q3 FY26?" — for that, the answer does not exist in this tool.

Company resolution: Company names are mapped to Yahoo Finance ticker symbols via TICKER_MAP in utils.py. HDFC → HDFCBANK.NS, Reliance → RELIANCE.NS.

Input:  "What is HDFC Bank's current P/E ratio?"
Output: {"Company": "HDFC", "Fundamentals": {"HDFCBANK.NS": {"Trailing PE": 18.4, ...}}}
historical_price_tool Historical Price Tool

Data source: Yahoo Finance OHLCV data via yfinance

What it answers: Open, High, Low, Close, Volume, and % change for a specific trading date.

Non-trading day handling: If the requested date is a weekend or holiday, the tool looks backward up to 7 days to find the nearest trading session and clearly reports which actual date was used.

Input:  "How did HDFC stock perform on 2026-01-15?"
Output: {"Requested Date": "2026-01-15", "Actual Trading Date Used": "2026-01-15",
         "Open": 1842.50, "Close": 1798.30, "Percentage Change (%)": -2.40, ...}
get_gnews_articles News Tool

Data source: GNews API — real-time news aggregation, filtered to India, English language

What it answers: Recent news, stock movement catalysts, regulatory updates, analyst actions, macro events affecting the company. Searches a 7-day window ending on the query date.

Returns: List of article dicts with title, description, URL, published_at, and source name. The agent formats these as headline + 1-2 line summary + hyperlink.

Input:  "Why did Reliance stock fall on 2026-01-15?"
Output: [{"title": "RIL Q3 results disappoint; EBITDA misses estimates",
          "url": "https://...", "published_at": "2026-01-15T09:31:00Z", ...}]

API & Frontend

7.1 FastAPI backend

The backend exposes a single meaningful endpoint: POST /ask. The agent is initialized once at startup (module level) and reused across requests.

EndpointMethodPurpose
/GETHealth check — confirms server is running
/askPOSTAccepts a {"query": "..."} body, runs the agent, returns answer + tools_used
/docsGETAuto-generated Swagger UI (built into FastAPI)

The /ask handler streams agent events internally — every tool call and LLM token is logged — while returning a single JSON response to the caller:

{"answer": "HDFC Bank's PAT for Q3 FY26 was ₹16,736 crore...",
 "tools_used": ["rag_tool"]}

7.2 Retry & rate-limit handling

Gemini's API can return 503 (Service Unavailable) during high load or 429 (Rate Limited) when request frequency exceeds quota. The /ask handler retries up to 4 times:

  • 503: Exponential backoff (2s, 4s, 8s)
  • 429: Parses the retry-after delay from the error message; adds a 2-second buffer. Falls back to exponential if no delay is specified.
  • Non-retryable errors: Returns HTTP 500 immediately with a safe error message

If the agent LLM fails but tools already returned data, the fallback returns the raw tool outputs directly — the user still gets partial information rather than a blank error.

7.3 Streamlit frontend

The frontend is a single-page chat interface that assembles the structured query from sidebar inputs and displays the agent's response.

UI ElementPurpose
Date picker (sidebar)Sets the Date: field — used by news and price tools for time-scoping
Company selectorCurrently HDFC / Reliance — maps to indexed vector stores
Financial Year selectorSets the Financial Year: field for RAG retrieval enrichment
Quarter selectorQ1–Q4 or None — fine-tunes RAG retrieval to specific quarter
Example question buttonsOne-click starters for common query types
Tools used badgesShows which tools the agent called (RAG, News, Price, Fundamentals)
Sources expanderLists the PDF filenames the RAG tool retrieved from

The frontend calls http://127.0.0.1:8000/ask with a 120-second timeout. Errors (connection refused, timeout, HTTP 500) are caught and displayed as friendly warning messages.

Source marker stripping

The rag_tool appends a [SOURCES_USED: file1.pdf; file2.pdf] tag to its response for the evaluation harness to parse. The frontend's parse_answer() function strips this marker before displaying the answer to the user.

Evaluation Framework

The system has two complementary evaluation modes, each measuring a different layer of the pipeline. Both run directly against the Python code — no running server needed.

8.1 End-to-end Agent Evaluation (evaluate.py)

Runs the full agent against 30 test cases from evaluation_sample.csv and measures how well the agent routes, retrieves, and responds.

MetricWhat it measuresFormula
Tools AccuracyDid the agent call exactly the right set of tools?exact_match / total_cases
Tools PrecisionOf the tools called, what fraction were correct?|called ∩ expected| / |called|
Tools RecallOf the expected tools, what fraction were called?|called ∩ expected| / |expected|
Keywords RecallDo expected terms appear in the response?found_keywords / total_expected_keywords
Multi-hop AccuracyDid the agent call multiple tools when required?(len(tools) > 1) == expected
Sources PrecisionOf RAG sources used, what fraction were expected?|used ∩ expected| / |used|
Sources RecallOf expected sources, what fraction were retrieved?|used ∩ expected| / |expected|
Latency (avg/p95/max)End-to-end query time in secondstime.perf_counter() around agent.run()

The test cases cover all four tools, single-hop and multi-hop scenarios, and questions for both HDFC and Reliance. Run with: make evaluate

8.2 Hallucination Evaluation (hallucination_eval.py)

Bypasses the agent entirely and evaluates the RAG generation sub-pipeline in isolation. For each RAG test case, it calls _retrieve_context and _generate_answer directly, then asks Gemini to act as a hallucination judge.

Retrieve context
(for test question)
→
Generate answer
(Gemini, temp=0.1)
→
Hallucination judge
(Gemini)
→
has_hallucination
+ claims list

The judge receives the retrieved context, the question, and the generated answer, then returns structured JSON:

{
  "has_hallucination": false,
  "hallucinated_claims": [],
  "verdict": "All claims in the answer are directly supported by the retrieved context."
}

This detects cases where the model confidently states numbers or facts not present in the retrieved chunks. Run with: make hallucination

Why not RAGAS? RAGAS is a popular RAG evaluation library that does similar things. We built custom evaluators instead because (1) we use Gemini, not OpenAI, and (2) the custom approach integrates directly with our pipeline internals (_retrieve_context, _generate_answer), giving us precise control over what exactly is being evaluated.

Performance Improvements

The following improvements were made to the original codebase. Each entry explains what changed, why it matters, and the impact.

ChangeWhat & WhereWhy it matters
LLM temperature = 0.1 agent_system.py — GoogleGenAI init Lower temperature reduces randomness in tool selection and response wording. Financial data should be deterministic — the same question should always route to the same tool.
RAG generation temperature = 0.1 rag_model.py — GenerateContentConfig Generation with temp=0 or ~0 prevents the model from "creatively" filling in numbers it didn't see in context. Critical for financial accuracy.
max_function_calls 5 → 10 agent_system.py — FunctionAgent Complex multi-tool queries (RAG + news + fundamentals) use 3 tools minimum. With a cap of 5, retries on 503/429 errors would cause silent failures. Raised to 10 to give headroom.
Enriched query passed to reranker rag_model.py — reranker.postprocess_nodes Previously, only the bare question was passed to the cross-encoder. The quarter/FY hints added during retrieval were discarded at reranking. Now the reranker sees the full enriched query, correctly penalizing wrong-period chunks.
BM25 node corpus persisted to disk + in-memory cache build_index.py + rag_model.py BM25 was rebuilt from scratch on every request (~0.5–1s). Now build_index.py saves the node corpus (text + metadata) to vector_store/{company}/bm25_nodes.json. At query time the JSON is loaded and BM25Retriever is rebuilt from those nodes in under a second, then cached in memory. Note: pickling the BM25Retriever object directly fails because pystemmer (used internally) contains Cython objects that cannot be serialized — JSON node storage is the correct workaround.
Removed context markers from agent output rag_model.py — rag_tool return The [RAG_CONTEXT_START]...[RAG_CONTEXT_END] block (all 6 raw chunks, ~3000 tokens) was appended to the tool result sent back to the agent. This bloated the agent's reasoning context on every tool call. Removed — the agent now only receives the synthesized answer + source names.
Removed contradictory rag_tool description tools.py — FunctionTool description The description simultaneously said "use for qualitative analysis" AND "do NOT use for exact numeric values". Since the financial_statement_tool was disabled, rag_tool is the only source for numbers — the description now clearly says so.
Removed dead code: eval_mode rag_model.py — rag_tool signature The eval_mode parameter was never used — both eval scripts bypass rag_tool and call _retrieve_context directly. Removed to simplify the function signature.
gnews error returns descriptive string gnews_tool.py — except block On API failure, the tool returned []. The agent interprets an empty list as "no news found" rather than "an error occurred". Now returns a descriptive error string so the agent can explain the failure to the user.
Improved system prompt — data not found agent_system.py — system_prompt Previously: just say "Data not available". Now: agent must explain which tools were searched, why data might be missing (e.g. quarter not indexed), and suggest what the user can do instead. Much more useful UX.
Latency metrics in evaluation evaluate.py, eval_utils.py Added avg, p95, and max latency per run. p95 is the most useful metric for performance benchmarking — it captures real-world slow cases without being skewed by a single outlier.
Fixed stale CSV test cases evaluation_sample.csv 5 test cases expected financial_statement_tool which was removed from the system. These were always failing, artificially deflating the tools_accuracy metric. Updated to rag_tool and replaced with specific Q4 FY26 questions.
Fixed inflated source expectations Both CSV files Q1 questions incorrectly listed Q1+Q2+Q3 PDFs as expected sources. Only Q1 source should be expected for Q1-specific questions. Fixed to correctly scope expected sources per quarter.
Module docstring placement rag_model.py The module docstring was placed after all imports. Python only recognizes a string at the very top of a file as the module __doc__. Moved to line 1.
Fixed duplicate import + inline import main.py FastAPI was imported twice. import re was placed inside an except block (ran on every error). Both moved to top-level imports.

How to Run

10.1 Prerequisites

  • Python 3.13+
  • A .env file in the project root with GEMINI_API_KEY and GNEWS_API_KEY

10.2 First-time setup

venv\Scripts\pip install --upgrade pip
venv\Scripts\pip install uv
venv\Scripts\uv pip install -e .

# Build vector indexes from PDFs in data/
venv\Scripts\python backend\build_index.py

10.3 Running the application

# Terminal 1 — start FastAPI backend (port 8000)
venv\Scripts\uvicorn backend.main:app --reload

# Terminal 2 — start Streamlit frontend
cd frontend
..\venv\Scripts\streamlit run app.py

Open http://localhost:8501 for the UI. API docs at http://localhost:8000/docs.

10.4 Running evaluations

# End-to-end agent evaluation (tool accuracy, keywords, latency) — 30 test cases
venv\Scripts\python -m evaluation.evaluate evaluation\evaluation_sample.csv

# RAG hallucination evaluation — 16 RAG test cases
venv\Scripts\python -m evaluation.hallucination_eval evaluation\rag_evaluation_sample.csv

10.5 Adding a new company

  • 1
    Create data/{company_name}/ and add earnings PDFs
  • 2
    Add the company's Yahoo Finance ticker to TICKER_MAP in backend/tools/utils.py
  • 3
    Run venv\Scripts\python backend\build_index.py to index the new documents
  • 4
    Add the company to the company selectbox in frontend/app.py
  • 5
    Add test cases to the evaluation CSVs
